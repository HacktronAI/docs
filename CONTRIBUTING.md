# Contributing

Thanks for helping improve the Hacktron docs. This guide covers the two
automated areas of the site ‚Äî the API Reference and product screenshots ‚Äî and
how to work with them. For general writing standards and local preview, see the
[README](README.md).

## API Reference (don't hand-edit endpoint pages)

The endpoint pages under **API Reference** are generated by Mintlify from
`api-reference/openapi.json`. Do **not** add or edit MDX files for individual
endpoints ‚Äî they would be ignored and drift from the spec.

- To change an endpoint's documentation, the change must land in the OpenAPI
  spec. The spec is refreshed automatically from production by the
  `sync-openapi` workflow.
- The curated **Getting started** pages (`introduction`, `authentication`,
  `rate-limits`, `pagination-filtering`, `errors`) **are** hand-written MDX and
  are the right place for narrative, conventions, and examples.
- Endpoints are referenced from `docs.json` as `METHOD /path`, and each
  generated page's URL is derived from the operation summary (for example
  `POST /scans`, summary "Trigger a pentest scan", renders at
  `/api-reference/scans/trigger-a-pentest-scan`). Link to that URL, not to a
  `.mdx` path.

Validate locally before opening a PR:

```bash
pnpm install
pnpm exec mintlify openapi-check api-reference/openapi.json
pnpm exec mintlify broken-links
```

## Screenshots

Screenshots are produced by the pipeline in `automation/`. Images live in
`images/` and are referenced from MDX with a `<Frame>`. Captures are **dark by
default** and mask PII; only the seeded test org is ever driven.

### The placeholder contract

Every screen that needs a visual but doesn't have one yet uses the shared
placeholder. When you add such a section:

1. Reference the placeholder in the page:

   ```mdx
   <Frame>
     <img src="/images/placeholder.png" alt="What this screen shows" />
   </Frame>
   ```

2. Add a manifest entry (see below) with `status: "todo"`.
3. Trigger the **Sync screenshots** workflow. The next run replaces the
   placeholder, points the entry at the captured image, and flips it to
   `ready`.

### Adding a new screenshot

Append an entry to `automation/screenshots.json`:

```json
{
  "name": "platform-new-screen-dark",
  "route": "/new-screen",
  "auth": true,
  "theme": "dark",
  "viewport": { "width": 1440, "height": 900 },
  "waitFor": "h1:has-text(\"New screen\")",
  "mask": ["[data-clerk-component=\"UserButton\"]", "[data-testid=\"user-email\"]"],
  "status": "todo"
}
```

Field notes:

- **`route`** ‚Äî path **without** the org slug for authenticated screens; the
  harness prefixes `E2E_ORG_SLUG` at runtime. Use the full path for public
  screens and set `auth: false`.
- **`waitFor`** ‚Äî a Playwright locator the harness waits on as the stability
  anchor. **Never** rely on network idle.
- **`mask`** ‚Äî locators for anything showing emails, names, or other PII/dynamic
  content; these are blacked out in the screenshot.
- **`status`** ‚Äî `"todo"` until the pipeline (or a human) fulfills it, then
  `"ready"`.

Then run the **Sync screenshots** workflow (`workflow_dispatch`) or wait for the
weekly schedule. The run opens a PR only if an image changed, and lists any
entries it could not produce.

### Running the harness locally

Requires the secrets listed in the [README](README.md#required-secrets) exported
into your shell.

```bash
cd automation/screenshots
npm install
npx playwright install --with-deps chromium

# Sign in once and save the authenticated storage state.
npm run setup

# Capture every manifest entry (only changed images are written).
npm run capture
```

`npm run placeholder` regenerates `images/placeholder.png` if you ever need to.
