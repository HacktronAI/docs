# Hacktron Documentation

Welcome to the Hacktron documentation repository! This repository contains the [official documentation](https://docs.hacktron.ai) for Hacktron, powered by [Mintlify](https://mintlify.com/).

Before contributing, please read this document carefully to understand our documentation standards and contribution process.

## Contributing

We welcome contributions from the community to improve our documentation, e.g. fixing typos, adding examples, or expanding explanations. To contribute, please follow these steps:

1. Fork this repository
2. Create a new branch for your changes
3. Make your changes in the appropriate `.mdx` files
4. Test your changes locally
5. Submit a pull request describing your changes

### Prerequisites

* Node.js 24

* Git

* A code editor (we recommend VS Code with MDX extension)

### Local Development

1. Install dependencies:

```bash
pnpm install
```

2. Start the development server:

```bash
pnpm dev
```

3. Visit `http://localhost:3000` to preview changes

4. Make changes to `.mdx` files and see them live-reload

### Environment Setup

For VS Code users, we recommend installing these extensions:

* MDX

* Prettier

* Tailwind CSS IntelliSense

## Writing Guidelines

### Document Structure

Each document should follow this structure:

```mdx
---
title: 'Document Title'
description: 'A brief description of the content'
---

## Overview

Brief introduction to the topic.

## Main Content

Your primary content sections.

## Related
- Link to related doc 1
- Link to related doc 2
```

### Component Usage

Use Mintlify components to enhance documentation:

```mdx
<Tabs>
  <Tab title="Example 1">
    Content for example 1
  </Tab>
  <Tab title="Example 2">
    Content for example 2
  </Tab>
</Tabs>

<Steps>
  <Step title="First Step">
    Instructions for first step
  </Step>
</Steps>

<Card>
  Important information here
</Card>
```

### Code Examples

* Use syntax highlighting

* Include comments and write idiomatic code

* Keep examples concise

* Test all code examples

Good example:

```python
def greet(name):
    """Return a greeting message."""
    return f"Hello, {name}!"
```

Bad example:

```python
def greet(name):
    return "Hello, " + name
```

## Automation

Two automated pipelines keep generated content fresh. Both are **PR-only** (they
never push to the default branch), idempotent (no PR when nothing changed),
concurrency-guarded, run with least-privilege permissions, and pin their GitHub
Actions to commit SHAs.

### API Reference (OpenAPI)

The **API Reference** endpoint pages are generated by Mintlify from
[`api-reference/openapi.json`](api-reference/openapi.json) — they are **not**
hand-written. The `openapi` field on the API Reference tab in `docs.json` wires
the spec into navigation, and each endpoint is referenced by `METHOD /path` so
the group order is preserved. The curated **Getting started** pages
(`introduction`, `authentication`, `rate-limits`, `pagination-filtering`,
`errors`) remain hand-written MDX.

[`.github/workflows/sync-openapi.yml`](.github/workflows/sync-openapi.yml) runs
daily (and on demand via **workflow_dispatch**). It fetches the **public**
spec, validates it with `mintlify openapi-check`, writes
`api-reference/openapi.json`, and opens a PR only when the spec changed. It only
ever reads the public `*/api-docs-json` endpoint (prod preferred) and **never**
the internal `/docs` basic-auth spec.

To regenerate or validate locally:

```bash
pnpm install
pnpm exec mintlify openapi-check api-reference/openapi.json
pnpm exec mintlify broken-links
```

> **Note on this initial PR:** the committed `openapi.json` was bootstrapped from
> the documented endpoints because the CI sandbox could not reach
> `api.hacktron.ai`. The first `sync-openapi` run replaces it with the canonical
> production spec.

### Screenshots

Product screenshots are produced by a manifest-driven, self-healing Playwright
pipeline under [`automation/`](automation/):

- [`automation/screenshots.json`](automation/screenshots.json) — the manifest.
  Each entry is `{ name, route, auth, theme, viewport, waitFor, mask, status }`
  (plus `usedBy`/`shows` notes). Authenticated `route`s are prefixed with the
  org slug at runtime, so the slug is never committed.
- [`automation/screenshots/`](automation/screenshots/) — the harness:
  `setup.mjs` (Clerk sign-in + Vercel bypass cookie → saved storage state),
  `capture.mjs` (per-entry capture, diff, and `todo → ready` self-heal), and
  `generate-placeholder.mjs`.

Stability rules baked into the harness: screenshots are captured **dark by
default** (matching `docs.json` `thumbnails.appearance` and the existing
`*-dark.png` files); the color mode is set deterministically via the Nuxt
`color-mode` localStorage keys + the browser `colorScheme` and then verified;
navigation **never** waits on `networkidle` (the app holds a LaunchDarkly stream
open) — every entry waits on its explicit `waitFor` locator. PII is masked at
capture time and only the seeded test org is driven.

[`.github/workflows/sync-screenshots.yml`](.github/workflows/sync-screenshots.yml)
runs weekly (and on demand). It installs Chromium, signs in, captures every
entry, and opens a PR only when an image changed. The PR body lists any
`todo`/failed entries (route unreachable, state not producible in CI) for a
human to supply.

> The `*.mp4` video and `/logo/*` assets are not screenshots and are not part of
> the manifest.

#### Required secrets

The human adds these in the repository's Actions secrets (none are hardcoded):

| Secret | Purpose |
| --- | --- |
| `CLERK_PUBLISHABLE_KEY` | Clerk test publishable key (`pk_test_…`). |
| `CLERK_SECRET_KEY` | Clerk test secret key (`sk_test_…`). |
| `E2E_USER_EMAIL` | Seeded test user email for programmatic sign-in. |
| `E2E_USER_PASSWORD` | Seeded test user password. |
| `E2E_ORG_SLUG` | Slug of the seeded test org; prefixes authed routes. |
| `VERCEL_BYPASS_TOKEN` | Vercel deployment-protection bypass, set as the `_vercel_jwt` cookie. |
| `DOCS_APP_BASE_URL` | Staging app base URL (defaults to `https://staging.hacktron.ai`). |

#### Placeholder contract

When the Mintlify Agent (or a human) adds a section that needs a visual:

1. Reference the neutral placeholder in a `<Frame>`:

   ```mdx
   <Frame>
     <img src="/images/placeholder.png" alt="Describe the screen here" />
   </Frame>
   ```

2. Add a manifest entry to `automation/screenshots.json` with `status: "todo"`
   (and the route/locator/masks for the screen).
3. Dispatch the **Sync screenshots** workflow (or wait for the weekly run). The
   next run captures the screen, replaces the placeholder image, and flips the
   entry from `todo` to `ready` — self-healing the section.

See [CONTRIBUTING.md](CONTRIBUTING.md) for the step-by-step on adding a new
screenshot and running the harness locally.
